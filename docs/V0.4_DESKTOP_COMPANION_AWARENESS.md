# Nexus v0.4 Series — Desktop Companion Awareness

v0.4 is the first major step after the v0.3 foundation line. It should make
Nexus feel less like a chat panel that waits for messages and more like a quiet
desktop companion who understands that time passes while the user works
elsewhere.

This is not a screen-control release, not a productivity-agent release, and not
an always-on recorder. It is the beginning of companionship continuity.

## Product Intent

When the user opens Nexus and then stops interacting with Nexus directly, that
does not mean the companionship ends. It means the companion should stay quiet,
notice coarse desktop activity when allowed, and understand that the user has
been going through a stretch of time.

The goal is for a later conversation or a low-frequency check-in to feel
continuous:

```text
You have been at this for about half an hour. Want to pause for a minute?
```

or:

```text
You have been switching between the project and the browser for a while. Want
me to help you collect the current thread?
```

Nexus should not speak like a stopwatch, surveillance tool, or task manager.
It should speak like a companion who was quietly present.

## Current Release Priority

`v0.4.0` is now the stable Quiet Observation Foundation release. `v0.4.1`
Coarse Time Language, `v0.4.2` Check-In Policy, and `v0.4.3` User-Facing
Transparency remain draft stacked follow-ups. The current active stacked
v0.4.x slice is `v0.4.4` Beta Feedback And Copy Tuning: normalize beta reports
into safe labels and harden check-in copy against surveillance-like or
timer-like language before expanding behavior.

Do not expand `v0.4.4` into external notifications, message sending, tool
execution, new sensing sources, feedback analytics, adaptive copy generation,
raw timelines, productivity scoring, pet motion, or desktop window control.
Those remain later v0.4.x or v0.5 work.

## Core Rules

1. Opening Nexus allows companionship, but does not immediately create noise.
2. If the user is interacting with Nexus, normal chat takes priority.
3. If the user leaves Nexus and keeps using the computer, Nexus may enter quiet
   observation when desktop awareness is enabled.
4. Observation produces coarse activity segments, not raw surveillance logs.
5. Time language is approximate: a while, about half an hour, about an hour,
   two hours or more.
6. The model receives sanitized companionship summaries, not raw screenshots,
   precise timers, full clipboard contents, or unredacted window text.
7. Proactive check-ins must be low-frequency, dismissible, and easy to pause.
8. The feature must never become a Codex-style autonomous work agent by
   default.

## Version Split

`v0.4.0` stable is complete. Each later v0.4 release should ship one layer of
the experience. Do not merge sensing, time language, proactive check-ins, and
history UI into one large release.

### v0.4.0 — Quiet Observation Foundation

Purpose: make Nexus understand "I am open, the user is elsewhere, time is
passing" without speaking yet.

- Track when Nexus is open and when the user last interacted with Nexus.
- Detect that the user is active elsewhere on the desktop.
- Maintain a short-lived local activity segment summary:
  - current broad activity class
  - rough duration bucket
  - whether the user is focused, switching often, or idle
  - whether Nexus should stay silent
- Expose a visible pause/resume control.
- Pass only a sanitized companionship summary into the chat prompt.

Must not do:

- no proactive notifications yet
- no pet movement changes
- no full activity history UI
- no raw screenshot, raw clipboard, or exact window-title prompt injection

Acceptance:

- When Nexus is open and unused, a local summary can say the user has been
  active elsewhere for a rough duration.
- When the user is chatting with Nexus, quiet observation does not compete with
  normal chat.
- The feature can be paused and the prompt summary disappears.
- Tests cover session start, last Nexus interaction, active-elsewhere
  detection, sanitized prompt shape, and pause behavior.

### v0.4.1 — Coarse Time Language

Purpose: make every user-facing time expression feel companion-like instead of
timer-like.

- Keep observation thresholds and trigger logic unchanged from `v0.4.0`.
- Move localized elapsed-time wording and precise-time detection into a thin
  companion time language layer.
- Normalize all elapsed-time language into buckets:
  - just started
  - a while
  - about half an hour
  - about an hour
  - two hours or more
- Avoid minute/second precision in user-facing companion copy.
- Add regression tests so prompts, UI, and stored recent summaries cannot say
  "37 minutes and 12 seconds", "1小时30分钟", "1時間30分", or "2시간 10분".

Must not do:

- no proactive scheduling changes yet
- no check-in policy expansion
- no analytics dashboard

Acceptance:

- All desktop-awareness summaries use the same time-bucket formatter.
- Prompt text and UI text do not expose exact minute/second durations.
- Locale copy has natural equivalents in zh-CN, zh-TW, en-US, ja, and ko.
- Tests cover bucket boundaries, localized labels, forbidden precise-time
  phrases, multilingual numeric-unit leaks, and fallback from malformed labels
  back to safe bucket wording.

### v0.4.2 — Check-In Policy

Purpose: decide when Nexus may gently speak after quiet observation.

- Decide when Nexus may gently check in:
  - long continuous activity
  - frequent switching
  - long idle after activity
  - user returns to Nexus after being away
- Respect quiet hours, cooldowns, user focus, and explicit pause.
- Keep every check-in optional and easy to dismiss.
- Separate decision from emission: deciding that a line is allowed must not
  directly schedule, render, persist, or repeat it.
- Add local emission guards:
  - active chat suppresses all check-in lines
  - stale return-to-Nexus signals expire
  - the same activity signal cannot repeat inside the same emission window
  - recently dismissed signals stay suppressed
  - generated in-app payloads are local, dismissible, and expire quickly

Must not do:

- no external notifications until the local in-app behavior is stable
- no message sending or tool execution
- no "you must rest now" language
- no repeated nagging inside the same activity segment
- no persistent check-in history or cross-session check-in carryover

Acceptance:

- Check-ins are rate-limited and explainable.
- User focus suppresses unnecessary interruptions.
- Returning to Nexus can surface a gentle continuity line without pretending to
  know private details.
- Active chat with Nexus takes priority over every check-in signal.
- Repeated polling cannot emit the same check-in line again inside the emission
  window.
- Dismissing a line suppresses the same signal for a local dismissal window.
- In-app payloads are passive data objects only: dismissible, expiring, and
  local to the current session.
- Tests cover cooldown, quiet hours, pause, focused activity suppression,
  frequent switching, long continuous activity, return-to-Nexus behavior,
  active chat suppression, stale return windows, duplicate emission windows,
  dismiss suppression, signal-key stability, and local in-app payload shape.

### v0.4.3 — User-Facing Transparency

Purpose: make desktop companion awareness understandable and controllable.

- Keep the current Settings layout, but route the companion awareness block
  through a small transparency view model so policy state and UI copy stay
  aligned.
- Add a structured transparency contract that exposes:
  - current companion awareness status
  - status label key
  - what Nexus can observe
  - what reaches the model
  - what is stored locally
  - whether a recent summary exists
  - why model reach is blocked
  - why clearing a summary is unavailable
  - the local storage lifetime kind
  - a hard `rawContentVisible: false` invariant
- Keep pause semantics explicit: paused awareness blocks model reach, but a
  user can still clear an already-existing recent local summary because that is
  a local deletion action.
- Add a lightweight invariant check around the view model so future refactors
  cannot silently drift from the privacy contract.
- Settings must explain:
  - what Nexus can observe
  - what it stores
  - what reaches the model
  - how to pause or delete recent activity summaries
- Show the current coarse companion awareness state without raw content.

Must not do:

- no raw activity timeline
- no productivity score
- no hidden export of desktop activity
- no new visual redesign of the Settings screen
- no new sensing source, model prompt expansion, or storage model change

Acceptance:

- Settings shows current status and the privacy boundary in plain language.
- Users can pause awareness and clear recent summaries.
- The UI never displays raw screenshots, private message bodies, or full
  clipboard content.
- The clear-summary action is disabled with an explicit local reason when no
  recent summary exists or awareness is unavailable.
- If a recent local summary already exists, clearing it remains available while
  awareness is paused.
- Model reach is explicit: off, paused, waiting for observation, or allowed by
  a current sanitized summary.
- The view model is deterministic and does not carry raw window titles,
  clipboard bodies, message bodies, file paths, screenshots, or precise
  minute/second time labels.
- Tests cover transparency status, row ordering, action state, blocked reasons,
  storage lifetime, invariant drift, raw-content leakage, and precise-time
  regression.

### v0.4.4 — Beta Feedback And Copy Tuning

Purpose: use real beta feedback before expanding behavior.

- Collect community beta reports about timing, tone, and interruption feel.
- Normalize beta feedback into irreversible safe labels:
  - interaction context
  - timing feel
  - tone feel
  - privacy-boundary signal
  - OS permission friction
- Keep feedback signals stateless and non-persistent; they must not train,
  personalize, score, or adapt behavior.
- Harden check-in copy so every locale stays caring, optional, and non-monitoring.
- Add locale-aware guardrails for gentle time-passing lines.
- If copy validation fails, fall back to a fixed safe companion line instead of
  silently dropping the in-app check-in.
- Document OS permission friction and known false positives.

Must not do:

- no feedback analytics engine
- no raw feedback text retention
- no cross-session feedback linking
- no adaptive or generated check-in copy
- no new sensing source unless beta feedback proves the current sources are not
  enough
- no v0.5 pet behavior yet

Acceptance:

- Community beta template covers v0.4-specific questions.
- The template includes structured safe labels for interaction context, timing,
  tone, privacy-boundary signals, and permission friction.
- The template asks whether Nexus was open but unused, whether active chat took
  priority, whether check-ins felt caring or like monitoring, and whether pause
  and clear controls were discoverable.
- Feedback normalization keeps only semantic labels and never retains raw
  comments, timestamps, window titles, clipboard bodies, screenshots, file
  paths, user IDs, or session traces.
- Feedback signals remain non-training, non-adaptive, non-scoring, local, and
  non-persistent.
- Copy tests cover every locale and every check-in reason, protecting against
  surveillance-like phrasing, imperative language, and exact timer language.
- Unsafe copy falls back to a static safe line and does not disappear silently.
- Release notes summarize the draft feedback and copy-safety boundary without
  switching the stable release entry point.

## Out of Scope for v0.4

- Mouse following and animated pet reactions. That belongs to v0.5.
- Automatic mouse/keyboard control.
- Full screen recording history.
- Precise productivity analytics.
- Reading files or messages without explicit authorization.
- Sending messages, opening apps, editing files, or changing settings without
  confirmation.

## Privacy Boundary

The v0.4 data path should preserve the v0.3 desktop-context privacy posture.

Allowed model-facing summary:

```text
Nexus has been open. The user has not interacted with Nexus for about half an
hour. Recent desktop activity looks like coding with occasional browser
reference checks. The user may be focused; avoid interrupting unless asked.
```

Not allowed:

```text
Exact raw window titles, full clipboard text, screenshots, OCR dumps, private
message bodies, file paths, precise second-level timers, or hidden activity logs.
```

## Acceptance Criteria

- The feature remains default-quiet.
- The user can pause desktop awareness from a visible control.
- Prompt summaries use rough time buckets only.
- Raw screenshots are never passed into chat context.
- Desktop activity summaries are redacted before model use.
- Tests cover time-bucket formatting, prompt summary shape, pause behavior, and
  privacy redaction.
- Release notes clearly state that v0.4 is companion awareness, not screen
  control or autonomous work execution.

## Relationship To Community Growth

The v0.4 series should be easy for the community to validate and extend without
touching private data. Good community contributions include:

- beta reports about whether check-ins feel too frequent or too cold
- language variants for gentle time-passing copy
- OS-specific notes for desktop awareness permissions
- safe test fixtures that reproduce false positives without private content
- persona examples that respond to "you have been at this for a while" without
  sounding like surveillance

Community work must remain reviewable, licensed, and local-first.
