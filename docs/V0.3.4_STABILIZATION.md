# Nexus v0.3.4 Stabilization Gate

This gate turns the current beta line from "feature landed" into "releaseable
on real desktops." Protocol tests are necessary, but they do not prove the
macOS Full Disk Access flow, real Telegram/Discord network behavior, or
user-visible recovery paths.

## Product Position

Nexus is currently optimizing for:

1. **Desktop presence**: the companion stays visible and lightweight.
2. **Real context**: foreground app, notifications, clipboard/screen/OCR where
   granted, and communication bridges feed the companion.
3. **Local-first continuity**: local model and local voice paths should remain
   viable, with cloud providers as explicit choices.
4. **Bounded initiative**: proactive care is event-driven, rate-limited, and
   explainable.

Do not expand into a character marketplace, generic AI workbench, or universal
private-message scraper while this gate is open.

## P0 Evidence Matrix

| Area | Required evidence | Command or manual check |
|---|---|---|
| Repo health | No unintended working tree changes before release prep | `git status --short` |
| Unit coverage | Existing bridge/webhook/storage regressions pass | `npm test` |
| Build health | TypeScript and Vite production build pass | `npm run build` |
| i18n health | Locale keys stay complete after copy changes | `npm run i18n:audit` |
| Docs alignment | README, ROADMAP, release notes no longer contradict v0.3.4 behavior | Read `README.md`, `docs/ROADMAP.md`, latest release notes |
| Local webhook injection | A synthetic `kind: "message"` payload is accepted by the real Nexus notification bridge and a local evidence report is saved | `npm run message:smoke:local` (or `npm run message:validate -- --token <settings-token> --evidence-file artifacts/v0.3.4/message-awareness-local.json` against an already-running instance) |
| macOS message awareness | With Full Disk Access granted, a real Notification Center message is observed once and not replayed | Enable Desktop message awareness, send a real WeChat/QQ/Telegram/Slack test notification, inspect chat/debug console |
| Telegram live bridge | Real Telegram DM pairing, owner-only routing, text reply, voice fallback, reconnect no-replay | BotFather token + pairing code + real DM test |
| Discord live bridge | Real Discord channel/DM pairing, owner/allowlist routing, text reply, audio attachment when configured, reconnect status | Discord bot token + Message Content Intent + real channel test |

If any live check cannot be performed in the current environment, mark it
`unknown`, not `done`.

## Local Webhook Validation

For a reproducible local smoke that does not depend on renderer settings, run:

```bash
npm run message:smoke:local
```

This starts a smoke-only Electron main process, starts the real
`notificationBridge` webhook on `127.0.0.1:47830`, injects one synthetic
`kind: "message"` payload, writes
`artifacts/v0.3.4/message-awareness-local.json`, and then terminates the smoke
process. If another Nexus instance is already running, close it first or use
the manual command below against that instance.

For a manual check against an already-running Nexus instance, start Nexus with
the notification bridge enabled, then run:

```bash
npm run message:validate -- --token "Bearer nexus_..." \
  --evidence-file artifacts/v0.3.4/message-awareness-local.json
```

After the live macOS / Telegram / Discord checks have been performed, attach
their machine-readable notes to the same report:

```bash
npm run message:live:template
```

The template command refuses to overwrite an existing file unless
`--force-live-template` is supplied. Prefer recording each live check through
the scenario recorder instead of editing JSON by hand. The recorder keeps
unrelated checks pending, timestamps the run, maps the right proof flags for
each live target, and still lets the strict validator reject `pass` or `fail`
entries that do not carry the required proof.

```bash
npm run message:live:record -- macos \
  --live-evidence-file artifacts/v0.3.4/message-awareness-live.json \
  --observed-at "2026-06-16T15:00:00-07:00" \
  --operator "Klein" \
  --app-name "WeChat" \
  --full-disk-access-granted \
  --notification-observed-once \
  --replay-checked-after-restart \
  --note "One real WeChat notification appeared once in Nexus after Full Disk Access."
```

For Telegram and Discord, use the matching scenario and boolean proof flags:

```bash
npm run message:live:record -- telegram \
  --live-evidence-file artifacts/v0.3.4/message-awareness-live.json \
  --operator "Klein" \
  --update-offset 401 \
  --last-outbound-at "2026-06-16T16:01:00-07:00" \
  --last-outbound-kind "text" \
  --last-outbound-target "REDACT_OR_DESCRIBE_TARGET" \
  --pairing-approved \
  --owner-text-reply-returned \
  --busy-message-queued-or-retried \
  --reconnect-replay-checked \
  --note "Owner DM paired, replied, queued while busy, and did not replay after restart."

npm run message:live:record -- discord \
  --live-evidence-file artifacts/v0.3.4/message-awareness-live.json \
  --operator "Klein" \
  --last-reconnect-at "2026-06-16T17:02:00-07:00" \
  --last-reconnect-reason "gateway_reconnect_requested" \
  --last-outbound-at "2026-06-16T17:03:00-07:00" \
  --last-outbound-kind "text" \
  --last-outbound-target "REDACT_OR_DESCRIBE_TARGET" \
  --message-content-intent-enabled \
  --approved-channel-reply-returned \
  --bot-echo-suppressed \
  --reconnect-status-visible \
  --note "Allowed Discord channel replied once, ignored bot echoes, and exposed reconnect status."
```

To record a real failure, set `--status fail` and include a concrete
`--note`. Keep a check as `manual-required` until its proof fields are true.
Passing live records must include a real observation timestamp, an operator,
and a concrete non-template note; the recorder supplies `observedAt`
automatically only when `--observed-at` is omitted, but it still requires the
operator and note to make the release evidence auditable.
`--update-offset`, `--last-reconnect-at`, and `--last-reconnect-reason` are
optional diagnostics copied from Settings -> Console -> Advanced diagnostics.
`--last-outbound-at`, `--last-outbound-kind`, `--last-outbound-target`, and
`--last-outbound-error` can also be copied from the gateway traces to show the
reply attempt that matched the live check. These strengthen the evidence trail
but do not replace the required proof booleans.

Before merging live notes into the local webhook report, audit the live evidence
file directly:

```bash
npm run message:validate -- \
  --check-live-evidence artifacts/v0.3.4/message-awareness-live.json
```

For a release gate command that fails until all three live checks are `pass`,
use the npm shortcut:

```bash
npm run message:gate:live
```

To inspect the current release-evidence state without printing private message
payloads, use the status report. It lists pending proof fields and next
commands, plus a per-scenario live verification checklist that points to the
expected diagnostics traces such as Telegram `update offset`, Telegram/Discord
`last outbound`, and Discord `last reconnect`. When those optional diagnostics
already exist in the live evidence file, the suggested record commands preserve
the private-safe values such as timestamps, update offsets, reconnect reasons,
and outbound kind. Outbound target/error values are represented only as present
flags in the status JSON so chat or channel identifiers are not printed. It is
not a release gate:

```bash
npm run message:status:release
```

Then merge the live notes into the local webhook evidence report:

```bash
npm run message:merge:release
```

Finally, audit the complete report. This recomputes the release gate from the
local webhook check plus all three live checks instead of trusting the stored
`releaseGateComplete` value:

```bash
npm run message:gate:release
```

`npm run prerelease-check -- v0.3.4` also enforces this gate for stable tags.
Because `artifacts/` may contain real message details and is intentionally
ignored, stable CI should receive a redacted copy through
`docs/release-evidence/v0.3.4-message-awareness.json` or
`NEXUS_MESSAGE_AWARENESS_EVIDENCE_FILE`.

Generate that commit-safe copy from the complete raw report:

```bash
npm run message:release:redact
```

This command refuses to write the redacted copy until
`message:gate:release` is green, so incomplete live evidence cannot be
published accidentally.

The redacted report preserves the release-gate booleans, timestamps, and safe
app-name proof while removing webhook message text, sender/chat identifiers,
webhook response ids, live-check operators, free-form notes, and outbound
target/error diagnostics that may contain private chat or channel identifiers.

Use `--dry-run` when Nexus is not running:

```bash
npm run message:validate -- --dry-run \
  --evidence-file artifacts/v0.3.4/message-awareness-dry-run.json
```

three live release gates:

Allowed live statuses are `pass`, `fail`, and `manual-required`. The CLI rejects
`pass` unless the check includes observed time, operator, concrete notes, and
its required proof fields exactly as `true` or a non-empty string. A `fail`
check must include `notes`. The generated gate report sets
`releaseGateComplete: true` only when local webhook injection and all three live
checks are `pass`.

Expected acceptance result:

- The command exits `0`.
- The webhook response is `ok` or another successful JSON response from Nexus.
- The generated payload has `kind: "message"`, stable `conversationId`, and
  stable `messageId`.
- In the app, the message appears as an external-message notice or companion
  event according to the current notification settings.
- The evidence JSON has `gate: "v0.3.4-message-awareness"`, marks local webhook
  injection as `pass`, and keeps macOS/Telegram/Discord live checks as
  `manual-required` unless a `--live-evidence-file` supplied stricter evidence.

This proves only the local ingress contract. It does not prove macOS
Notification Center access or Telegram/Discord live behavior.

## macOS Message Awareness Validation

Use a real app notification, not a unit-test fixture.

1. Install or run Nexus on macOS.
2. Grant Full Disk Access to the running Nexus app.
3. Enable Desktop message awareness in settings.
4. Keep message preview reading disabled unless validating that path.
5. Send one real notification from a supported app such as WeChat, QQ,
   Telegram, Discord, Slack, Teams, Mail, DingTalk, or Feishu/Lark.
6. Confirm one and only one Nexus event appears.
7. Restart Nexus and confirm old notifications are not replayed.

Record:

- app name,
- whether notification previews were visible at the OS level,
- Nexus event timestamp,
- whether the companion spoke source-only or source+preview,
- any skip reason or permission error shown in the app.

## Telegram Live Validation

Minimum pass:

1. New bot token configured.
2. Empty allowlist rejects strangers or waits for pairing.
3. First owner DM receives a one-time pairing code.
4. Approving the code in desktop settings writes the numeric chat ID.
5. Owner text message enters the companion chat.
6. Companion reply returns to the same Telegram chat.
7. A message sent while the assistant is busy is queued or retried, not lost.
8. Restarting the gateway does not replay old updates.

The Settings -> Console -> Advanced diagnostics Telegram row exposes the
current `update offset` checkpoint after updates are confirmed, plus the most
recent outbound text/voice reply target and any send error. Use the offset as
supporting evidence for the reconnect no-replay check: the offset should not
reset to `0` across reconnects. Use the outbound trace as supporting evidence
for the owner-reply-returned check; it does not replace the manual check that
the reply appeared in Telegram.

Voice-specific pass:

- Incoming Telegram voice note is transcribed when API STT is configured.
- `inbound` voice reply mode replies with voice only after the user sent voice.
- Local WAV-only TTS falls back to text for Telegram voice bubbles.

## Discord Live Validation

Minimum pass:

1. Bot token and Message Content Intent configured.
2. Empty allowlist rejects strangers or waits for pairing.
3. Approved channel or user ID receives companion replies.
4. Bot-authored gateway echo does not re-enter the assistant as a new user
   message.
5. Gateway reconnect status is visible after network interruption.

The Settings -> Console -> Advanced diagnostics Discord row exposes the most
recent outbound text/audio reply target, send errors, and reconnect
reason/time. Use those traces as supporting evidence for reply-returned,
audio-attachment, and reconnect-status checks; still confirm the reply appears
in the real Discord client.

Audio-specific pass:

- Discord audio reply attaches a playable file when a supported TTS provider is
  configured.
- Discord inbound voice remains documented as not transcribed unless that path
  is implemented.

## P1 Follow-On After P0

Only after the P0 matrix is green or honestly marked with known gaps:

1. Keep expanding the desktop-context diagnostics card. The first version now
   lives in Settings -> Console -> Advanced diagnostics and reports foreground
   window, clipboard, screen/OCR, Notification Center, webhook, Telegram, and
   Discord readiness without starting or stopping bridges. It also exposes the
   local ingress command, live-evidence template, live gate, private-safe
   release status report, merge command, and full release gate commands.
   Telegram and Discord outbound replies now leave a "last outbound" diagnostic
   trace with target, kind, timestamp, and send error when one occurs. Discord
   reconnect attempts now surface as a "last reconnect" diagnostic trace with
   reason and attempt count, so the reconnect-status live check has a visible
   place to verify. Telegram now surfaces the Bot API update offset as a
   no-replay checkpoint for reconnect validation, and the offset is persisted
   per bot token so app restart does not fall back to offset `0`.
   Telegram/Discord messages that also appear through the macOS Notification
   Center watcher now share a cross-pipeline announcement key, and local
   native-bridge echoes are suppressed when the matching native bridge is
   enabled. Next evidence to add: longer-running live timestamps and richer
   skip/error causes from each bridge.
2. Keep proving proactive-care observability. The first local event log now
   records fired/skipped/error outcomes for away pings, daily brackets, open
   arcs, and future capsules, and the latest decisions are visible in Settings
   -> Console -> Advanced diagnostics. It can now copy a bounded public
   evidence report with fired/skipped/error counts, quiet-hours skips,
   rate-limit skips, source-link coverage, openable source-route coverage, total
   time-window coverage, key decision-window coverage for due-item, quiet-hours,
   and rate-limit behavior, per-source openable source-route coverage, and the
   latest event categories. The public report omits event ids, details,
   source-ref ids, source-ref labels, and non-code reason strings, while keeping
   bounded next actions for failed evidence checks. The same report can be
   regenerated directly from the local Electron storage with
   `npm run proactive:care:evidence -- --local-storage-leveldb "$HOME/Library/Application Support/nexus/Local Storage/leveldb" --output artifacts/v0.3.4/proactive-care-evidence.json --require-ready`.
   Settings -> Console -> Proactive care can copy the local event-row export
   for fallback use as `artifacts/proactive-care-events.json`; keep that export
   local and attach only the generated redacted evidence report when evidence
   must be shared.
   `npm run proactive:care:evidence -- --sample` can sanity-check the
   private-safe report path with synthetic rows and is marked `sample-qa`; the
   stabilization status gate rejects sample QA artifacts, so release evidence
   still has to use exported runtime events.
   Source refs are now actionable: message refs jump to the retained message in
   Settings -> History; scheduler and bracket refs route to Settings ->
   Autonomy; errand, arc, and capsule refs route to the Autonomy care queues and
   highlight the matching object when that object still exists. Scheduler-only
   refs now also show a concrete source detail card for the relevant Autonomy
   controls or queue. Next evidence to add: longer real-run samples across
   quiet-hours, cooldown, and due-item windows.
3. Keep expanding memory ownership UI. The first version now lets users browse,
   edit, forget, pin, pause recall, and source-trace long-term memories, plus
   edit/forget daily entries. Chat-derived long-term memories and chat/voice
   diary source refs now jump to the matching saved message in Settings ->
   History when the message is still retained; scheduler, bracket, errand, arc,
   and capsule memory refs can now route to Settings -> Autonomy using the same
   source-target detail path as proactive-care diagnostics. Relationship-shaped
   memories and generated reflections now have a separate Settings -> Memory lane
   with the same ownership controls. Reflection edit recall now has a
   private-safe local evidence helper and focused test: it compares before/after
   ranking deltas and metadata preservation without copying memory text, memory
   ids, source refs, topics, or queries. Next evidence to add: longer-running
   real-user samples that exercise the same path through the Settings editor.
4. Keep expanding Live2D and voice control. The first Live2D action map editor
   can save per-model settings overrides. The same action-map coverage can now
   be regenerated as private-safe evidence with
   `npm run live2d:action-map:report -- --model mao --output artifacts/v0.3.4/live2d-action-map.json`,
   including optional override patches without copying exact expression names,
   motion group names, or fidget stage directions. The first voice diagnostics
   panel copies private-safe timing reports for wake/mic, STT, first-audio
   latency, first-audio budget status and recommendation, and the active TTS
   provider's request policy (`delta-stream` vs `round-buffer`, first chunk,
   and request-size caps). The same evidence can be regenerated with
   `npm run voice:diagnostics:report -- --input artifacts/voice-diagnostics-input.json --output artifacts/v0.3.4/voice-diagnostics.json --require-ready`;
   the public report omits transcript preview, pipeline detail, trace text/ids,
   transition session/provider/reason/meta fields, and TTS model/voice ids.
   Settings -> Console -> Voice diagnostics can copy the local input export
   used as `artifacts/voice-diagnostics-input.json`; keep that input local and
   share only the generated public report.
   Playback now uses the same provider policy for first-audio watchdog
   timeouts, giving delta-stream providers a tighter failure window while
   keeping round-buffer providers tolerant of slower startup. Character Card
   imports now expose a safe import report covering
   profile/persona files, pending greeting, and lorebook normalization without
   copying full card content. The same report can be generated without writing
   the imported profile via
   `npm run character:card:report -- --sample --output artifacts/v0.3.4/character-card-import.json`;
   replace `--sample` with `--card-file ./card.json` when validating a real
   private card locally.
   They also import the first safe Nexus role package preset fields from card
   extensions: style anchors, local/keyless voice overrides, tool allow/block
   lists, and pet model ids; API keys and base URLs are ignored and the report
   records only booleans and counts. Active character profiles now expose an editable preset panel for
   avatar and keyless/local voice defaults, syncing edits back to the current
   settings draft. `voxtral-local` and `kyutai-local` are now registered as
   keyless local OpenAI-compatible TTS adapters, and the readiness report can
   distinguish "candidate registered" from "active target engine selected with a
   measured first-audio sample."
   `npm run tts:adapter:smoke -- --provider voxtral-local --output artifacts/v0.3.4/tts-adapter-smoke.json --require-ready`
   or the same command with `--provider kyutai-local` now runs a private-safe
   localhost `/audio/speech` smoke that records first-byte latency, byte count,
   and budget status without copying text/model/voice values once a local
   adapter service is running. Next evidence to add: deeper package workflows
   and real runs against a local adapter service.

The same Advanced diagnostics section also has a private-safe P1/P2
stabilization evidence package. It aggregates context diagnostic coverage,
proactive-care observability, and voice latency checks while omitting webhook
tokens, outbound targets, provider error text, voice transcripts, and raw trace
details. This package is supporting evidence for the P1/P2 upgrade plan, not a
replacement for the P0 message-awareness release gate above.
Use `npm run stabilization:evidence:status` to index the file-backed P1/P2
evidence artifacts in `artifacts/v0.3.4` without copying artifact contents into
the status report. The status report expects proactive-care, Live2D action-map,
Character Card import, voice diagnostics, and local TTS adapter smoke artifacts;
`--require-ready` exits non-zero until all five are present and `ok=true`.

P3 adapter groundwork is limited to the webhook contract. Use
`npm run message:adapter:check -- --template email` or
`npm run message:adapter:check -- --template im` to generate a starter payload,
then validate the filled payload with `npm run message:adapter:check` before
wiring an email or additional-IM adapter. The checker reports payload shape,
dedupe readiness, support boundary, permissioned capture method, and privacy
warnings without sending a webhook or copying private sender/text/id values
into the report. Add
`--output artifacts/v0.3.4/message-adapter-<source>.json --require-ready` to
persist the same private-safe JSON as adapter-readiness evidence and fail the
command when required contract checks are not ready. Planned email / IM adapters
must declare a capture method such as
`mail-rule`, `imap-api`, `public-api`, `system-notification`,
`user-automation`, or `export-file`; private app database scraping is blocked by
the contract. It does not make email, WeChat, QQ, DingTalk, Feishu/Lark, Slack,
or Teams a native bridge; those remain permissioned adapter work after the P0
live gate is green.
