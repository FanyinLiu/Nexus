## 1. CURRENT STATE

The "通讯桥" is not one system. It is **three disjoint inbound pipelines** (Telegram gateway, Discord gateway, notification webhook/RSS) that never converge, plus **three half-built outbound paths** (auto-reply: never wired; manual reply: blocked by default; reminder broadcast: ungated). Inbound mostly works; outbound is where it has "一直搞不定".

### Telegram gateway (main process) — **working** (inbound), with sharp edges
`electron/services/telegramGateway.js` + `electron/ipc/telegramIpc.js`. Module-global singleton: `getMe` token verify, then `getUpdates` long-poll; allowlisted messages fan out to all windows as `telegram:message` (telegramIpc.js:8-12); `sendMessage` IPC exists and works. Verified edges: empty allowlist accepts **every** chat (`if (_allowedChatIds.size > 0 && ...)`, telegramGateway.js:91); non-text messages silently dropped ("Skip non-text messages for now", :98); `_updateOffset = 0` on every connect (:144) so any reconnect — now routine since fb1f5eb made credentials reactive — can replay the last batch and offline backlog as duplicates; poll errors keep `state: 'connected'` forever and retry every 3s with no backoff or cap (:113-125), so a revoked token never surfaces as an error state. No 429/rate handling anywhere.

### Discord gateway (main process) — **working** (inbound), different failure shape
`electron/services/discordGateway.js` + `discordIpc.js`. Hand-rolled Gateway WS v10 (heartbeat/identify/resume), REST send. Verified: `connectGateway` resolves on WS `'open'` (:272-275) before READY sets `'connected'`, so the IPC connect result reports `'connecting'`; after 10 reconnect attempts it goes terminally `'error'` until the user toggles the integration (:296-300); malformed-frame guard from fb1f5eb is in place (:277-285). Asymmetric vs Telegram on every axis: string channelId vs number chatId, carries `fromUserId` (Telegram shape doesn't — why owner detection differs per channel), loses messages during downtime (no backfill) where Telegram replays them, has reconnect where Telegram has none.

### Connection lifecycle (renderer hooks) — **partial**: inverted ownership
`src/hooks/useTelegramGateway.ts` / `useDiscordGateway.ts` (near copy-paste twins). Connections live inside a renderer `useEffect` keyed on `[enabled, token, allowlist]` — the fb1f5eb fix made credential edits reconnect, but the structural consequence stands: the bridge exists only while the panel renderer tree is mounted; reload/close/crash tears it down, main only force-disconnects at quit. No headless receive. The `status` object the hooks return is consumed by no UI; `refreshStatus` has zero callers; preload pushes no status-changed event — a dead connection is invisible except via the settings-page probe, whose refresh effect's dependency array omits the telegram/discord fields (`IntegrationsSection.tsx:104-115`).

### Inbound bridge controllers — **working** (this half is solid)
`src/app/controllers/useTelegramBridge.ts` / `useDiscordBridge.ts` (~95% twins). Verified flow: owner detection via CSV match (useTelegramBridge.ts:56-61) → optional announcement → injection into the single companion chat as `【Telegram】` (owner) or `【Telegram · Name】` (stranger) via `void chat.sendMessage(..., { source: 'telegram' })` (:86-91). The system prompt in all 5 locales teaches the model this convention. Per-chat reply targets tracked in refs; every inbound chatId is remembered for broadcast (:96).

### The outbound/reply loop — **partial → effectively broken**: the core gap
Three facts, all verified:
1. **The assistant's reply never returns to the channel.** `source` is branched on exactly once in the entire chat pipeline — `source === 'voice'` (useChat.ts:479); in `assistantReply.ts` it feeds only logging/TTS metadata. `chat.sendMessage`'s result is discarded (`void`, useTelegramBridge.ts:90). The model has no send tool — `builtInToolSchemas.ts` defines only `web_search`/`weather`/`open_external`.
2. **The only reply path is dead by default.** `replyTo` checks `isActionAllowed(settings, 'telegram', 'send')` (useTelegramBridge.ts:130); default mode is `'confirm'` (settings.ts:162,169); confirm+send returns `allowed: false / needs_confirmation` (permissions.ts:69-75); `needs_confirmation` has **zero** consumers and **no settings UI exposes either permission mode** (grep of `src/components` is empty). The block writes a debug-console line and silently returns (:131-137).
3. **The manual reply UI fakes success.** `PanelView.tsx:286-291` does `await replyToTelegram(...)` then unconditionally marks the notification read and clears the input — `replyTo` returns `undefined` both on permission block and on unresolvable target, so a blocked/no-op reply looks sent. And the UI only appears for NotificationMessages whose `sourceId` is literally `telegram`/`tg`/`discord` (:256-262) — i.e. webhook-relayed items, never actual gateway messages — and resolves targets against the **gateway-populated** chat map, so it works only when the gateway coincidentally saw the same chat.

Plus: inbound bridge messages are **silently dropped while a turn is busy** — `useChat.ts:506-516` returns false with recovery UX only for `fromVoice`; telegram/discord get no queue, no retry, no record.

### Permission system — **stub**
`src/features/integrations/permissions.ts`. Graduated read-only/confirm/auto exists on paper, but: confirm has no confirmation flow, no UI can change modes, enforcement is renderer-only (`telegram:send-message` / `discord:send-message` IPC handlers in main do no permission check), and `broadcastToChannels` bypasses it entirely (verified: no `isActionAllowed` anywhere in coreRuntime.ts — sole import site is the bridge hooks). Net policy is incoherent: replies are blocked by default, broadcasts are auto by default.

### Reminder broadcast — **partial**, the only autonomous outbound path
`coreRuntime.ts:248-286` `broadcastToChannels` sprays reminder text to every known Telegram chatId / Discord channelId; the known sets are in-memory module Sets seeded from allowlists **plus every inbound sender** — so with the default-open allowlist, any stranger who ever messaged the bot receives the owner's private reminders. Sole caller: reminder 'notice' tasks (useReminderController.ts:88). No throttle, no per-target selection, state lost on reload.

### Notification bridge (webhook + RSS) — **partial**
`electron/services/notificationBridge.js`. Localhost HTTP server on hardcoded 127.0.0.1:47830 with a 0600-persisted bearer token; SSRF-guarded RSS polling; solid payload normalization. Verified gaps: channel config is module memory only (`let _channels = []`, :27; the only `writeFile` in the file is for the token, :83) — RSS/webhook channels evaporate on restart; port-bind failure silently nulls the server; `'calendar'` kind is declared in types but dropped by the sanitizer; and `kind:'message'` payloads are **announce-only** — they become companion notices, never chat turns, and (except telegram/discord-labeled ones) can never be replied to. Heaviest churn in the whole stack: 9 commits, 6 of them security/hardening fixes.

### External adapters + CLI — **working**, strictly one-way
`scripts/send-message-webhook.mjs` + `scripts/communication-adapters/*` (macOS Notification Center watcher, Shortcuts, Hammerspoon, PowerShell). This is the de-facto WeChat/QQ/DingTalk/Feishu strategy: OS-notification scraping → webhook → announce. Shipped in v0.3.2 (d9ff3d2), tested, asar-unpacked. But announce-only ingest with no reply, run manually outside the app, and the macOS watcher also forwards Telegram/Discord notifications — so the same message can arrive twice via two pipelines with no cross-pipeline dedupe.

### Autonomy ↔ channels — **no connection**
The v2 decision engine outputs only `silent | speak | idle_motion`; letter/away/bracket schedulers use OS notifications. The companion **cannot** autonomously message anyone on any channel — the presumable heart of the v0.4.0 vision. The autonomy plumbing (quiet hours, focus awareness, scheduling) exists and is healthy; it lacks only an outbound action kind and a router.

### Credential storage — **working**, no debt
safeStorage vault + per-sender vault-ref resolution (keyVaultBridge.ts:8-16, vaultRefs.js:52-75). The one clean layer.

### Tests — **partial**: a real wall
`tests/gateways.test.ts:1-7` states it plainly: discordGateway/telegramGateway `import { net } from 'electron'` and cannot load under plain `node:test`. Both gateways, both IPC files, the inbound routing hooks, and the replyTo gate have **zero tests**. The identically-patterned but testable minecraftGateway needed three recent race fixes (b5ae6f0, bbd2e58, a4b88f7) — strong evidence the untestable twins harbor the same class of bugs undetected. Announcement building, allowlist parsing, and the webhook CLI are well-tested.

### Email — **dead** (does not exist)
No SMTP/IMAP/email code anywhere in `src/` or `electron/`; only a form-field type union mentions the word. Greenfield.

### Unmerged branch `feat/v0.3.4-message-voice` — salvageable, misnamed
Exactly one commit ahead of main: `79434d0` (Telegram media-message announce + pure `routeTelegramMessage()` extraction, +13 tests, 5-locale i18n). Verified: **merges clean into current HEAD** (`git merge-tree --write-tree` exits 0). Contains zero voice work despite the branch name.

## 2. WHY IT NEVER GOT NAILED DOWN (ranked root causes)

**#1 — The loop was designed but the closing half was never built (architectural).** `replyTo` was created on day one with a doc comment saying the chat runtime would call it; it had zero callers for two months until the manual PanelView textbox (2026-06-07). Meanwhile the system prompt instructs the model to "reply to the external contact," zh-CN settings copy promises 「双向通信」/「适合不在电脑旁时继续对话」, and FEATURES.md:211-214 claims "bidirectional messaging" and "Reply Routing | Companion replies sent back to originating chat/channel" (verified — both rows present). Every layer *assumes* a reply router that no commit ever implemented. The bridge ships as half a bridge while all surrounding text describes a whole one — so it permanently *feels* broken.

**#2 — Compounding silent failures make even the existing half feel unreliable (observability).** Five independent silent-failure points stack: permission block → debug console only; PanelView → fake success (marks read, clears input); busy assistant → inbound message vanishes; Telegram poll death → state stays 'connected' forever; gateway status → no UI consumer, no push event, stale probe. A user testing the bridge cannot distinguish "blocked by default permission," "dropped because busy," "connection silently dead," and "reply never implemented" — they all look identical: nothing happens. 「总觉得不可靠」 is the designed-in outcome.

**#3 — Three pipelines accreted instead of one abstraction (architectural).** Three incompatible message shapes, three IPC event names, twin hooks/controllers/services/IPC files, the `'text'|'voice'|'telegram'|'discord'` union duplicated across 9+ files, hardcoded `【Telegram】` prefixes mirrored in 5 locale prompts, per-channel switch in permissions.ts. Webhook channels are second-class (announce-only, no reply); gateway channels skip the notification store the reply UI reads. Each new channel costs ~6 layers / ~15 files, and behavior drift (reconnect policy, owner detection, media handling, backlog semantics) is structural. The history shows the announce seam alone was refactored three times (useAutonomyController → useTelegramBridge → telegramAnnouncement.ts → telegramMessageRouter.ts, the last unmerged) — repeated re-plumbing of the same inbound seam, never the missing outbound one.

**#4 — The test wall froze the gateways (testability).** Because both gateways import electron's `net`, they have zero tests and every change is hand-verified — verified comment at gateways.test.ts:1-7. The testable sibling (minecraftGateway) needed three race fixes; the untestable twins are riskier to touch, so since birth (3ec0e1f, 2026-04-11) they received only security hardening: SSRF, webhook auth, frame-parse guard, credential reload (18ae062 → f55f8de/8fe465f → cea924b/2fe2788 → fb1f5eb). **Zero commits in two months added outbound capability.** The bridge consumed its maintenance budget being made safe, never being made whole.

**#5 — Inverted lifecycle ownership caps the ceiling (architectural).** Main holds the sockets but the renderer decides when they exist. No headless receive, teardown on window reload — incompatible with "companion you can message anytime," so every incremental improvement still hits this wall.

**#6 — Strategy ambivalence (process).** ROADMAP.md keeps the bridges "experimental, not main path"; EXECUTABLE_OPTIMIZATION_TASKS.md Sprint 1 makes 消息抓取闭环 a P0 (3 of 4 tasks unchecked, no summarization code exists); FEATURES.md sells it as shipped. Three documents, three different commitments — no design got the sustained investment a closed loop needs.

**#7 — Setup friction suppressed the feedback loop (UX).** 7-9 steps across two apps, two near-identical CSV fields (allowlist vs owner), no chat-id self-discovery (the bot knows the chatId on first message but doesn't offer it), default-open allowlist, and misconfiguration detectable only as "怎么没反应". Low real usage → gaps never got pounded on → never prioritized.

## 3. WHAT A v0.4.0 BRIDGE MUST SOLVE (requirements checklist, not design)

Bounded by Klein's standing constraints: **stability-first / no big-bang refactor; file-based per-entity config over JSON-blob settings fields; soft user-facing wording; `npm run build` green every commit.**

**Loop closure (the actual feature)**
- [ ] Assistant replies to channel-sourced turns are routed back to the originating (channel, conversation) automatically — the send must be a first-class step in the reply pipeline, not a side effect, with its result observed (no `void`-and-forget).
- [ ] A model-facing outbound capability (tool or pipeline hook) so the companion can *initiate* sends — required for the autonomy vision (proactive messages); autonomy needs only a new action kind once an outbound router exists.
- [ ] Define what a "conversation" is: per-(channel, conversationId) threads or source-tagged turns in one stream — but chosen explicitly; today external contacts interleave with the master's thread and trigger desk-local TTS/bubbles/mood as if the master were present.

**One message model**
- [ ] A single normalized inbound message shape and a single conversation registry shared by gateway-native channels AND webhook-relayed channels — the reply affordance must key off the registry, not `sourceId === 'telegram'` string matching.
- [ ] Webhook `kind:'message'` items must be able to reach the AI (today they are announce-only) — or the limitation must be explicit and worded softly in UI.
- [ ] Cross-pipeline dedupe (same Telegram message arriving via gateway and via the macOS notification watcher) and cross-reconnect dedupe (Telegram offset replay).

**Policy in one place, honest by default**
- [ ] One permission gate covering reply, broadcast, and proactive sends — `broadcastToChannels` bypassing the gate while manual replies are blocked is the current contradiction.
- [ ] Enforcement in the **main process** at the IPC/send boundary, not only in renderer hooks.
- [ ] Either build the confirm flow (`needs_confirmation` currently has zero consumers) or drop 'confirm' from the mode set; either way, expose the mode in settings UI. A mode no UI can change is not a setting.
- [ ] Empty allowlist must mean deny (or require an explicit "accept anyone" opt-in) — default-open plus remember-every-sender plus ungated broadcast is the standing privacy incident shape.
- [ ] Reminder/proactive sends target explicitly chosen destinations, not "every chat ID seen this session."

**Lifecycle owned by main**
- [ ] Connections resident in the main process, configured from settings read in main — surviving renderer reloads, enabling headless receive.
- [ ] One shared connect/reconnect/backoff/status state machine (today: 4 gateways × 4 bespoke machines; minecraftGateway's generation-counter pattern is the proven reference).
- [ ] Truthful states: poll failure must leave 'connected'; reconnect exhaustion must be recoverable without a manual toggle; status pushed to renderer (event, not dead polling).
- [ ] Persist what must survive restart: notification channel config, known conversations — per Klein's preference, as files per entity, not JSON blobs in settings.

**Reliability**
- [ ] Inbound queue: messages arriving while a turn is busy are held, not dropped.
- [ ] Rate limiting + 429 handling on every outbound path; sequential broadcast needs throttling.
- [ ] No fake success: every send path reports delivered/blocked/failed, and the UI reflects it — in soft wording ("didn't make it out, kept for retry"-style, not "Failed").

**Testability**
- [ ] Gateways loadable under plain `node:test` — inject `net`/fetch/WebSocket rather than importing electron at module top. This is the precondition for everything else; the minecraft race-fix history is the proof of need.
- [ ] Pure routing seams (the `routeTelegramMessage()` extraction in 79434d0 is the template) for inbound routing, owner detection, reply-target resolution, permission gating.
- [ ] Shared payload schema validation for messaging IPC (games already have payloadSchemas.js; messaging uses three ad-hoc idioms).

**Truth reconciliation (cheap, do first)**
- [ ] Fix FEATURES.md:211-214 and the zh-CN 「双向通信」 copy to match reality until reality catches up.
- [ ] Resolve the ROADMAP-vs-EXECUTABLE_OPTIMIZATION_TASKS contradiction: is the bridge main-path for v0.4.0 or not — explicitly, before code.
- [ ] Decide merge-or-drop for `79434d0` (it merges clean today; it bit-rots otherwise).

**Email**
- [ ] Greenfield: zero existing code, and the webhook contract is inbound-only so it is not an extension point for sending. Email must be a first consumer of the unified adapter shape, not another bespoke pipeline — that's the test of whether the abstraction works.

## 4. SALVAGE LIST

**Reuse as-is (healthy)**
- Credential vault: `src/lib/keyVaultBridge.ts`, `electron/services/vaultRefs.js`, per-sender vault-ref resolution — no debt.
- Webhook server + token auth + SSRF-guarded RSS: `electron/services/notificationBridge.js` (needs channel persistence added, core is sound).
- Payload normalizer: `electron/services/notificationBridgeUtils.js` — the closest thing to a normalized message shape already in tree.
- Adapter ecosystem: `scripts/send-message-webhook.mjs` + `scripts/communication-adapters/*` + `docs/COMMUNICATION_MESSAGE_WEBHOOK.md` — shipped, tested, the domestic-IM ingress story.
- Generic announcement builder: `src/app/controllers/messagingAnnouncement.ts` — privacy-defaulted, deduped, 5-locale, well-tested; the most bridge-ready abstraction in the codebase.
- Allowlist parsing: `src/features/integrations/allowlists.ts`.
- Commit `79434d0` (merge it: clean merge verified; brings media handling + the `routeTelegramMessage` testable-seam pattern + 13 tests).
- Lifecycle reference: `minecraftGateway.js` generation-counter/backoff pattern — battle-tested template for the shared state machine.
- System-prompt bridge sections (5 locales) — the owner/contact convention is fine; it just needs a runtime that honors it.

**Reuse with modification**
- Telegram/Discord protocol cores (`telegramGateway.js`, `discordGateway.js`): the Bot-API and Gateway-v10 protocol code is competent — keep it, but wrap in the shared lifecycle, inject network deps for testability, fix offset/backlog semantics, add 429 handling.
- Permission model (`permissions.ts`): keep the mode taxonomy; build the confirm consumer, move enforcement to main, expose in UI.
- `bridgeUtils.ts` reply-target resolution: keep the per-conversation map idea; rekey to the unified registry.
- `IntegrationsSection.tsx` panels + `integrationRuntime.js` probe: keep the surface; fix the probe dependency array, feed it pushed status.

**Replace / retire**
- Twin renderer hooks + twin bridge controllers (`useTelegramGateway`/`useDiscordGateway`, `useTelegramBridge`/`useDiscordBridge`) — collapse into one adapter-driven path; they are the copy-paste cost center.
- Renderer-owned connection lifecycle — invert to main-resident (this is the one place "no big-bang" needs care: it can be done one channel at a time behind the existing IPC surface).
- `broadcastToChannels` + in-memory known-ID Sets (coreRuntime.ts:149-172, 248-286) — replace with gated, targeted outbound through the unified router.
- PanelView reply plumbing (`getNotificationReplySource` string matching + unconditional success at PanelView.tsx:286-298) — rebuild on the registry with real send results.
- Dead code to clear when touched: `refreshStatus` (zero callers), `chat.bridge.telegram_announcement_*` i18n keys (defined in all locales, consumed nowhere), unconsumed gateway status returns from `useAutonomyController.ts:270-273`, the declared-but-sanitized-away `'calendar'` channel kind.
- FEATURES.md Messaging Gateways rows 211-214 and the 「双向通信」 settings copy — rewrite to match shipped reality immediately.
