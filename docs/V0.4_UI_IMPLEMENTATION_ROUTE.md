Purpose: turn the current Pro/open-source UI review queue into a local implementation route for Nexus 0.4. This document is a planning gate, not a visual spec by itself.

## Hard Rule

No raw Pro answer becomes code. Pro output must be converted into a recorded Nexus contract first, then implemented as the smallest surface-specific change that respects this route.

Reference projects are used for constraints, hierarchy, and behavior. They are not a license to clone another product's skin, palette, spacing, shell, dashboard, or component demo.

Before a Pro answer becomes intake, run `npm run ui:references:audit -- --surface=<surface> --pro-answer-quality`. Before any code change for that surface, run `npm run ui:references:audit -- --surface=<surface> --implementation-readiness`.

Generating a Pro payload is not the same as asking Pro. Before sending to the browser, run `npm run ui:references:audit -- --surface=<surface> --pro-send-payload`, use `npm run ui:references:audit -- --reference-refresh-check` only as the manual pre-send freshness check when the review depends on current open-source heads, paste only the prompt after user confirmation, then mark the surface sent with `npm run ui:references:audit -- --surface=<surface> --pro-registry-transition=sent`.

## Implementation Order

1. Contract freeze.
   - Confirm every 0.4 surface is recorded or accepted-for-prototype in the registry before implementing that surface.
   - If a surface is accepted-for-prototype, only the smallest surface-scoped prototype is allowed; final UI acceptance still requires a recorded/accepted contract.
   - If any surface is still not-sent, sent, intake-needed, needs-follow-up, or rejected, do not start implementation for that surface.
   - Keep `npm run ui:references:audit -- --pro-readiness` green before visual work starts.
   - Before asking Pro for a surface, run `npm run ui:references:audit -- --reference-refresh-check` manually when reference freshness matters; do not add it to default CI, release, lint, or build gates.
   - Keep `npm run ui:references:audit -- --surface=<surface> --pro-send-payload`, `npm run ui:references:audit -- --surface=<surface> --pro-answer-quality`, and `npm run ui:references:audit -- --surface=<surface> --implementation-readiness` in the workflow for each surface.
   - Do not start a UI change from a raw chat transcript, screenshot, or one-off preference without mapping it to a surface.

2. Image4 shell and rhythm.
   - Stabilize the companion panel regions first: header, Live2D stage, conversation recap, composer.
   - Keep the selected Image4 preview as a human browser review target.
   - Preserve the composer reachability and avoid using control resizing as an alignment fix.
   - Weather, date, and time remain secondary context and must not restore the retired dial or compete with Live2D.

3. Companion tone and color.
   - Warm-day should read as light, soft, low-arousal companionship, not as a dark workbench or cold admin surface.
   - Peach/apricot is a restrained emotional warmth accent; sage or mist-blue supports trust, focus, and send-ready states without taking over the whole field.
   - Do not sample reference project colors, clone palettes, add detached mood-light overlays, or make morning/day default dark.
   - Keep `npm run image4:color:audit` green before and after any companion-tone change.

4. Composer and input controls.
   - Align textarea, mic, send, and attachment centers as one input system.
   - Keep tool icons visually inside the input without persistent button backplates.
   - Focus clarifies the active control, but must not lift, resize, or reframe the whole composer.

5. Chat and streaming.
   - Keep normal messages readable and low-noise.
   - Add streaming, partial, resumable, tool-result, and final-answer states without turning every message into a card.
   - The composer remains stable and reachable during waiting or streaming states.

6. Settings, forms, and focus.
   - Group settings by trust boundaries: safety, memory, desktop awareness, integrations, and appearance.
   - Keep rows compact and repeated; avoid nested cards, provider-studio chrome, and dashboard sections.
   - Form controls use the right primitive: toggle for binary, segmented choice for modes, slider/stepper/input for numbers, menu for option sets.
   - Keyboard focus must remain visible through drawer navigation, tabs, message actions, and dangerous actions.

7. Agent activity and desktop awareness.
   - Agent activity means companion state, not autonomous task execution.
   - Allowed language: idle, context available, preparing reply, speaking, done, needs confirmation, blocked.
   - Use coarse time language such as just started, about half an hour, about an hour, or two hours or more.
   - Avoid terminal, task board, cockpit, file diff, precise minute-second tracking, or surveillance wording.

8. Human browser review.
   - Keep the Nexus panel open while tuning UI.
   - Check normal, narrow, short-height, focus, hover, disabled, speaking, waiting, and reduced-motion states.
   - Compare against the selected preview image only as a Nexus-specific fidelity target, not as permission to import unrelated chrome.

## Cross-Surface Guardrails

- Borrow constraints, not skins.
- Companion-first beats dashboard or workbench framing.
- No card stack, nested cards, cockpit, terminal log, timeline, or autonomous task board as default UI.
- Do not make headers, buttons, or utility controls larger to fix visual hierarchy.
- Text must fit within its own container on narrow and normal panel widths.
- Motion should clarify state only: speaking can animate the existing signal, focus can identify the current control, and hover can clarify affordance.
- Reduced motion must stay readable without replacing the design with a separate skin.
- Theme changes must be scoped to the surface being changed.

## Acceptance

0.4 UI work is ready to implement only when:

- the Pro review queue has no pending surface;
- the target surface has a recorded local contract;
- the target surface was sent to Pro only through a user-confirmed pre-send payload workflow;
- the target surface has passed the Pro answer quality checklist and implementation-readiness gate;
- the change names the surface it touches;
- the implementation avoids raw Pro output and exact open-source skin copying;
- the browser review confirms scale, alignment, containment, and focus states.
