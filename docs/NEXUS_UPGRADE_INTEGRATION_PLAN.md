# Nexus 升级整合计划

> 目标：把 Nexus 从“功能很多的桌面 AI 项目”收敛成一个方向清楚、阶段明确、可以持续升级的 **AI 桌面伴侣 + 自主数字助手**。

## 核心判断

Nexus 的长期方向是对的：桌宠形象、语音、记忆、本地模型、自动执行任务，这些能力组合起来才像一个住在电脑里的 AI 伙伴。

但当前最容易失控的点也很明显：Live2D、语音、记忆、Agent、MCP、插件、多 provider、桌面感知都很有吸引力，如果一开始同时推进，项目入口会变得很重，用户和贡献者都不知道应该先看什么。

所以升级策略不是“继续加功能”，而是：

1. 先把第一阶段最小闭环做稳定。
2. 把高级能力从默认体验里降权或隐藏。
3. 每一阶段只接回一个核心能力。
4. 每次升级都必须形成可演示、可测试、可回退的闭环。

执行优先级与验收指标统一到
[可执行优化任务清单](EXECUTABLE_OPTIMIZATION_TASKS.md)。

## 产品北极星

**Nexus 是一个住在电脑里的 AI 伙伴。**

它的长期体验应该是：

- 它有一个轻量的桌面形象。
- 它能自然对话，之后能语音对话。
- 它记得用户真正重要的事。
- 它优先使用本地模型和本地数据。
- 它可以在明确授权后帮用户执行任务。

它不是：

- 普通聊天软件。
- 多模型 API 控制台。
- 单纯桌宠动画。
- 通用 Agent 平台。
- 外部角色复刻。

## 总体整合策略

### 1. 不删除高级能力，先改成“后续能力”

仓库里已有语音、Live2D、记忆、自主、工具、Agent、MCP 等模块。短期不建议大规模删除，因为这些都是未来方向。

但它们不应该抢第一阶段的入口。处理方式：

- 默认 UI 不展示复杂高级设置。
- 高级模块保留代码，但通过设置、feature flag 或实验入口关闭。
- README 和 roadmap 明确说明：第一阶段只验证最小桌面伙伴。
- `FEATURES.md` 这类完整功能清单保留，但标注“长期能力/实验能力/已存在模块”，避免读者误以为第一阶段要全部完成。

### 2. 先做一条最短主路径

第一阶段只保留一条主路径：

```text
启动应用
  -> 显示桌面小窗口
  -> 显示极简头像
  -> 配置 Ollama 或 DeepSeek 文本模型
  -> 输入一句话
  -> 本地模型返回回复
```

只要这条链路稳定，Nexus 的“电脑里的 AI 伙伴”概念就成立了。其他能力都应该围绕这条链路逐步增强。

### 3. 每一阶段只整合一个大能力

升级顺序应该是：

1. 先让它存在。
2. 再让它有形象。
3. 再让它能说话。
4. 再让它记住。
5. 再让它能做事。

不要反过来先做完整 Agent、完整语音系统或完整 Live2D。

### 4. 竞品对标后的计划优化版（v0.3.2+）

上一版计划在结构上够清晰，但执行上要更聚焦“可用差异化”。对标同类后，优化方向是：

1. 把桌面感知和消息感知从“实验功能”改为“默认入口”。
2. 把模型接入从“参数堆叠”改为“一键可用”。
3. 把高级能力从默认路径移除，改为“明确授权后的可测闭环”。

优化执行计划：

- P0（2 周）：主路径 + 消息抓取闭环
  - 将当前窗口上下文（窗口标题/前台 app/截图）与通知 webhook 串成统一入口。
  - 让用户可快速执行：`Alt/Option + 空格` 打开小命令框，支持“询问当前窗口”和“汇总未读消息”。
  - Phase 0 新增验收：模拟 1 条消息可在 5 秒内形成桌面摘要，不崩溃。
  - 指标：P95 通知播报延迟 < 4 秒；未读摘要命中率（1 分钟内显示）> 90%。

- P1（4 周）：配置摩擦降到最低
  - 增加模型配置向导：选择区域（国内/国外）→ 厂商 → URL/API Key/模型，自动做一次可达性和鉴权检查。
  - 把常见 key 失败原因做可读化映射（中文、空格、备注、换行导致的 Header 无效）。
  - 把 MiniMax Token Plan、DeepSeek、OpenAI、Gemini、Claude 的入门模板放入设置向导。
  - 指标：首次通过率 ≥ 85%；第一次连接失败平均重试次数 ≤ 2；主路径 30 分钟内可完成一次成功会话。

- P2（6 周）：让伴侣行为可感知、可中断、可追溯
  - 宠物状态机加入“播报中 / 分析中 / 等待确认 / 已完成”的真实状态。
  - 未读消息卡片提供“稍后处理 / 标记重要 / 一键起草回复”动作。
  - 引入风险分级权限（一次性、临时、长期）并按粒度记录审计日志。
  - 指标：每个自动动作都有日志留痕；关键动作误触率下降（新用户误触率 < 5%）。

- P3（8 周）：分发与维护可信度并行增强
  - 统一未签名安装文案与故障提示，收口首轮安装失败路径。
  - 优先做体积和启动压力优化：模型/重资源延后加载、首屏资源裁剪。
  - 每月一次文档回看：按 [DOCUMENTATION_CONSISTENCY](DOCUMENTATION_CONSISTENCY.md) 确认 ROADMAP、README、FEATURES 与升级计划一致性检查通过。
  - 指标：一次完整发布前的发布门禁和文档同步不超过 24 小时延期。

P0-P3 的执行原则：每个阶段只接入能完成闭环的能力，不接入口即退化的“演示功能”。

## 阶段计划

### Phase 0：方向收敛和文档对齐

目标：让任何人打开仓库，先理解 Nexus 是什么、现在只做什么、未来再做什么。

工作项：

- 重写根 README，明确产品定位和第一阶段边界。
- 新增本文档，作为升级整合计划。
- 在 `docs/ROADMAP.md` 顶部增加链接，说明当前短期开发以 Phase 1 最小闭环为准。
- 梳理 `FEATURES.md`，把功能分成“Phase 1 必需”“后续核心”“实验能力”。

验收标准：

- README 前 30 秒能讲清 Nexus 是什么。
- 文档不再让人误以为第一阶段要完成 Live2D、Agent、语音、记忆全套系统。
- 开发者知道下一步应该先做桌面小窗口、极简头像、Ollama / DeepSeek 文本链路、简单对话。

### Phase 1：最小桌面伙伴

目标：完成最小可用闭环。

范围：

- 桌面常驻小窗口。
- 二次元极简头像。
- Ollama / DeepSeek 文本模型连接。
- 简单文字对话。

建议拆分：

| 模块 | 工作内容 | 复用方向 |
|---|---|---|
| 桌面窗口 | 小窗口、置顶、拖动、关闭、位置记忆 | 复用 `electron/windowManager.js`，收敛入口行为 |
| 头像形象 | 静态头像、idle/thinking/error 少量状态 | 复用 `src/features/pet` 的容器，先提供非 Live2D fallback |
| 文本模型连接 | Ollama 的 Base URL / 模型名，或 DeepSeek 的 API Key / 模型名；健康检查、错误提示 | 复用现有 chat/provider 能力，先绕开复杂 provider 切换 |
| 对话界面 | 输入、发送、回复、基础上下文 | 复用 `src/features/chat`，限制为 Phase 1 文本链路简化路径 |
| 设置入口 | 优先保留 Ollama / DeepSeek、模型名、必要密钥、窗口行为 | 暂时隐藏复杂 provider 矩阵、语音、工具、记忆设置 |

当前落地状态：

- 默认宠物预设切到 `nexus-mini`，用静态头像先跑通陪伴感，Mao/Live2D 保留为可选能力。
- 新安装的桌宠窗口默认使用更小的常驻尺寸，适合先放在桌面角落里运行；用户仍可拖拽调整。
- 新安装默认使用 Ollama 文本链路：`http://127.0.0.1:11434/v1` 和 `qwen3:8b`；已有 DeepSeek API Key 时可切换到 DeepSeek + `deepseek-v4-flash`。
- 设置首页收敛为常用、记录、进阶三组：模型、桌面、角色、自检、聊天记录、语音、记忆和工具；设定集合并进记忆，本地任务、集成和自治不再作为独立入口。
- 模型选择默认走 DeepSeek、Ollama、OpenAI-compatible 和自定义接口主路径，其他 provider 保留在更深的来源筛选里。
- 首次引导只保留文本模型和基础伙伴设置，不把连续语音作为 Phase 1 决策项。
- 默认聊天和小窗入口以文字对话为主；语音按钮只在进阶设置开启语音输入或已有语音会话时出现。
- Ollama / DeepSeek 连接测试已开始输出更明确的缺模型、缺 API Key、模型名不匹配提示。
- 自检集中显示预览地址、Electron 桥接、Ollama / DeepSeek 文本模型、头像和语音范围是否符合当前最小闭环。
- 新增 `npm run doctor`，可在命令行检查依赖、`electron:dev`、`47821` 预览服务、Ollama API 和 DeepSeek 配置入口。
- 本地网页预览使用 Vite 端口，例如 `http://127.0.0.1:47821/`；`11434/v1` 只用于 Ollama API。

Phase 1 不做：

- 完整 Live2D。
- 语音输入和语音输出。
- 长期记忆。
- Agent 循环。
- 工具调用。
- MCP、插件、游戏、聊天网关。

验收标准：

- 启动应用后能看到一个桌面小伙伴。
- 用户能配置 Ollama 地址和模型，或 DeepSeek API Key 和模型。
- 用户能输入一句话并收到本地模型回复。
- Ollama 未启动、模型不存在、DeepSeek 密钥缺失或请求失败时有清楚错误提示。
- `npm run lint`、`npm test`、`npm run build` 可通过。

### Phase 2：形象存在感

目标：让 Nexus 不只是聊天窗口，而是真正有“桌面伙伴”的存在感。

工作项：

- 增加头像状态：idle、thinking、speaking、error。
- 增加点击、拖拽、悬停等轻量互动。
- 增加非常简单的待机动作或表情切换。
- 保持 Live2D 可选，不作为默认必需能力。
- 定义 Nexus 自己的视觉方向：电脑里的 AI 伙伴，不复制外部角色。

验收标准：

- 用户不发消息时，Nexus 也像一个在桌面上待着的伙伴。
- 状态变化能反映“在思考/在回复/出错”。
- 形象轻量，不明显增加性能负担。

### Phase 3：基础语音

目标：在文本对话稳定之后，再加入最小语音闭环。

建议顺序：

1. 先做手动语音输入，不做唤醒词。
2. 再做 TTS 播放回复。
3. 再做打断和连续对话。
4. 最后考虑常驻唤醒词。

第一版语音只要求：

- 用户按按钮说话。
- Nexus 转成文字。
- 本地或配置的模型回复。
- Nexus 用一个可用 TTS 读出来。

暂不追求：

- 完整语音系统。
- 多 STT/TTS provider failover。
- 复杂声线系统。
- 常驻监听。
- 情绪化语音。

### Phase 4：可控记忆

目标：让 Nexus 开始记住用户，但必须可见、可编辑、可删除。

建议顺序：

1. 手动保存记忆。
2. 记忆列表。
3. 编辑和删除。
4. 对话时注入少量高价值记忆。
5. 再考虑自动提取和长期整理。

第一版记忆不做：

- 情绪系统。
- 关系等级。
- 梦境循环。
- 自动长期人格变化。
- 用户无法理解的隐藏记忆机制。

### Phase 5：自主数字助手

目标：让 Nexus 在用户授权后能帮忙做小任务。

第一批任务建议非常保守：

- 提醒事项。
- 简单搜索。
- 打开链接。
- 整理用户粘贴进来的文本。

权限原则：

- 读文件、写文件、联网、运行命令、控制应用都必须分级授权。
- 每次任务要有可读的计划和结果。
- 自动执行默认关闭。
- 高风险工具必须确认。

第一版不做：

- 完整 Agent 平台。
- 任意命令执行。
- 插件市场。
- 长时间无人监督任务。

### Phase 6：完整伴侣体验

目标：在前面闭环都成立后，再整合完整 Live2D、长期关系、主动陪伴、多模型和多端能力。

可接回能力：

- 完整 Live2D 模型和表情映射。
- 更自然的语音体验。
- 长期记忆和关系状态。
- 主动关心和低频提醒。
- 多 provider 和 failover。
- MCP 和插件能力。
- Discord、Telegram 等外部通道。

前提：

- Phase 1 到 Phase 5 的默认体验已经稳定。
- 高级能力都有明确开关、权限说明和失败处理。
- 不会让新用户一打开应用就面对复杂配置。

## 现有代码整合建议

### 应用入口

当前可优先围绕这些区域收敛：

- `electron/windowManager.js`：桌面窗口、托盘、置顶、显示隐藏。
- `src/app/views/PetView.tsx`：桌宠主视图。
- `src/features/pet`：头像和 Live2D 相关能力。
- `src/features/chat`：对话链路。
- `electron/ipc/chatIpc.js`：主进程聊天请求入口。

建议建立一条 `minimal companion` 路径，先让它只依赖窗口、头像、Ollama / DeepSeek 文本链路和文本对话。

### 高级模块处理

短期建议默认隐藏或降权：

| 模块 | 短期处理 | 回归阶段 |
|---|---|---|
| Live2D 完整系统 | 保留，但默认使用静态头像 fallback | Phase 2 / Phase 6 |
| Voice/STT/TTS | 保留设置和代码，默认不进入首屏 | Phase 3 |
| Memory/Relationship | 保留模块，先不默认注入复杂记忆 | Phase 4 |
| Agent/Autonomy | 保留实验能力，默认关闭 | Phase 5 |
| MCP/Plugins | 保留高级入口，必须 gated | Phase 5 / Phase 6 |
| Discord/Telegram/Game | 标记实验，不进入主路径 | Phase 6 以后 |

### 配置分层

建议把设置分成三层：

1. **基础设置**：Ollama / DeepSeek 文本链路、模型名、必要密钥、窗口行为、头像。
2. **伙伴能力**：语音、记忆、轻量主动提醒。
3. **高级实验**：Agent、MCP、插件、桌面感知、外部网关。

默认只展示基础设置。后两层可以通过“高级/实验”开关进入。

## 第一批开发任务清单

优先级从高到低：

1. 文档对齐：README、升级计划、ROADMAP、FEATURES。
2. 明确默认启动路径：打开后先进入桌面小窗口和简单对话。
3. 增加或确认静态头像 fallback，不依赖 Live2D 才能跑。
4. 做 Ollama / DeepSeek 健康检查和模型配置提示。
5. 简化首屏设置，只保留 Phase 1 必要项。
6. 给高级模块加明确实验入口或开关。
7. 增加 Phase 1 smoke test：窗口可启动、Ollama / DeepSeek 配置缺失时不崩、发送失败有错误提示。

## 风险和控制

| 风险 | 表现 | 控制方式 |
|---|---|---|
| 范围膨胀 | 一边做 Phase 1 一边加入语音、记忆、Agent | 每个 PR 标注所属阶段，Phase 1 只收最小闭环 |
| 首次启动过重 | 用户一打开就遇到 provider、语音模型、权限弹窗 | 第一阶段不请求麦克风、屏幕录制等高级权限 |
| Live2D 阻塞 MVP | 模型加载、资源路径、打包问题拖慢主体验 | 默认静态头像，Live2D 后接 |
| 文本模型配置失败 | Ollama 本地服务未启动、模型不存在、DeepSeek 密钥缺失或接口不兼容 | 健康检查、清楚错误、提供示例配置 |
| 高级模块干扰 | Autonomy、Agent、工具后台运行造成不稳定 | 默认关闭，只有用户主动启用才运行 |

## 推荐里程碑

### M1：最小演示版

结果：

- 小窗口出现。
- 极简头像出现。
- 输入文字后走 Ollama 或 DeepSeek 返回回复。

### M2：可用开发版

结果：

- 设置可配置 Ollama / DeepSeek。
- 错误提示完整。
- 基础上下文可用。
- 高级功能默认隐藏。

### M3：形象增强版

结果：

- 头像有状态变化。
- 桌面存在感更清楚。
- 轻量互动可用。

### M4：语音试验版

结果：

- 手动语音输入和 TTS 输出打通。
- 不做唤醒词和复杂连续对话。

### M5：助手能力试验版

结果：

- 提醒、搜索、打开链接等低风险任务可授权执行。
- 有任务日志和确认机制。

## 最终原则

Nexus 的升级不是把所有 AI 能力都塞进一个桌面壳里，而是不断增强这个核心体验：

**一个住在电脑里的 AI 伙伴，先能出现、能对话，再逐步能说话、能记住、能帮忙。**
