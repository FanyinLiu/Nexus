## 1. 现状关键事实（详证据见 BRIDGE_SURVEY）

- **三条 inbound 管道互不相通**：Telegram long-poll、Discord WS v10 进 chat pipeline（以 `【Telegram】` 前缀注入唯一伴侣 chat）；webhook/RSS 只播报不触达 AI。
- **outbound 缺口精确位置**：chat pipeline 对 `source` 只有一处行为分支（`fromVoice`，useChat.ts:479）。`assistantReply.ts` runner 全程持有 source，最终回复文本在 :461-466 定稿、:499 提交——source 与最终文本同时在作用域，是天然 seam，但没有任何 outbound 回调。回传函数 `useTelegramBridge.replyTo`（:112-143）已存在，唯一消费者是 PanelView 手动回复框。
- **连带 bug**：assistant busy 时 telegram/discord 消息无声丢弃（useChat.ts:506-517 只给 fromVoice 恢复 UX）；手动回复 UI 假成功（PanelView.tsx:286-291 无条件 mark-read，而 replyTo 权限拦截/target 不可解析时返回 undefined）；回复入口只对 webhook 中转项出现，gateway 原生消息不进通知 store（永久错配）。
- **安全债**：空 allowlist 默认放行所有人（telegramGateway.js:91、discordGateway.js:243）；主进程 send 无权限检查（grep isActionAllowed electron/ 零命中）；`broadcastToChannels`（coreRuntime.ts:248-286）绕过权限、会记住每个 inbound 发送者并向其直发提醒；`needs_confirmation` 权限模式零消费者、无设置 UI。
- **协议层 bug**：Telegram 每次 connect 重置 `_updateOffset = 0`（:144）→ 重连重放旧批次；poll 报错后状态仍 'connected'、3 秒裸重试无上限（:113-125）；Discord WS open 即 resolve（READY 才真连上）、10 次重连耗尽进终态 error 需手动 toggle、掉线期间消息丢失（与 Telegram 重放语义相反）。
- **重复度**：useTelegramGateway/useDiscordGateway（124/127 行）、useTelegramBridge/useDiscordBridge（146/142 行）、telegramIpc/discordIpc（48/46 行）三对 ~95% 孪生；每渠道需 6 层文件；source union 散布 9+ 处；`【Telegram】` 前缀硬编码镜像在 5 个 locale 系统提示。
- **可测试性墙**：telegram/discordGateway 顶层 `import { net } from 'electron'`（各自 :9）无法在纯 node:test 加载；网络触点各只有 1-2 个收口（apiCall 内 net.fetch + Discord 的 new WebSocket），其余逻辑已是纯 JS——注入开口明确。
- **健康项**：token 走 safeStorage vault（keyVaultBridge + vaultRefs 链路无债）；allowlists.ts 解析本身健康（去重、上限 256、格式校验）。
- 邮件渠道为全绿地（`grep -rniE 'smtp|imap|email'` 仅命中表单字段类型 union）。

## 2. 架构模式（业界调研结论）

调研对象：mautrix bridgev2、matterbridge、grammY/Telegraf/Botpress/Bot Framework、Chatwoot/Zendesk 会话模型、OpenClaw 安全模型、Delta Chat（chat-over-email）。来源 URL 见文末。

### 2.1 骨架：matterbridge 形态 —— 中心 Hub + 最小 ChannelAdapter

```ts
interface ChannelAdapter {
  connect(): Promise<void>
  disconnect(): Promise<void>
  send(msg: OutboundMessage): Promise<string>   // 返回平台消息 ID（写映射表）
  // inbound：统一推给 hub 的单一入口（异步队列方向）
}
```

- 方向不对称是业界共识：**出站同步方法调用**（错误传播、返回值、背压），**入站异步统一队列**。
- Hub 是普通 TS 类 + 构造函数注入，**不用全局事件总线**（EventEmitter 的 error-crash/无背压/流向难追踪问题有公认文献；VS Code 的分工范本：依赖走 DI，状态通知才用 typed Emitter）。
- 选它的理由：与现有三条管道一一对应。telegram/discord 协议核心（apiCall、WS 状态机）整体保留，要动的是每对孪生的 hook + controller 两层（~270 行/渠道）+ IPC 事件名归一 + main 侧 send 收口。

### 2.2 三个必抄细节（mautrix 的本质复杂度，不是过度设计）

1. **消息 ID 映射表**（远端消息 ID + part ↔ 本地消息 ID）：回复引用、将来编辑回复都绕不开。
2. **出站回声去重**：Discord WS 会把自己发的消息推回来，没这层 AI 自言自语。
3. **入站幂等键**：Telegram long-poll 语义是 at-least-once（崩溃后最多重放 100 条），按 update_id 去重是标配——顺便根治 offset 重置重放 bug。

### 2.3 会话模型：抄 Chatwoot

`Contact`（人）/ `ContactInbox`（人在某渠道的身份）/ `Conversation`（永远 per-channel）。**会话层永不跨渠道合并，身份层只靠显式信号合并**——没有任何主流系统默认把 Telegram 和 Discord 的同名用户猜成一个人。单用户应用里"主人"恒为 1 个 Contact，等价于"会话分渠道、AI 记忆全局共享"，几乎零成本。

### 2.4 安全模型：抄 OpenClaw

- **默认 deny**：空 allowlist = 拒绝所有人（现状的"空集放行"是调研确认的反面教材，n3d1117/chatgpt-telegram-bot 默认 `*` 被点名）。
- **配对码流程**：陌生 DM 触发配对 → 桌面端出一次性码（≤1h 过期）→ 渠道里回码 → **批准动作发生在已信任的桌面端** → 落成数字 ID allowlist（用数字 id 不用 username，username 可改名/抢注）。首个批准者成为 owner。
- 未授权消息**静默丢弃 + 本地日志**（不向探测者确认 bot 活着）。
- 平台事实：Telegram bot username 全局可搜、privacy mode 不管私聊；Discord 同服务器用户都能 DM bot，单主人正解是校验 author.id。

### 2.5 统一消息模型

- Canonical 层：matterbridge 式扁平 struct（13 字段量级）——消息 ID、渠道/账号、会话引用、发送者(id+名+头像)、CommonMark 正文、回复引用、附件列表(URL+mimetype+size+caption)、时间戳、event 枚举 + `extra` 逃生舱。
- **逃生舱原则**（BF Activity spec）：平台私货只准附加、不准成为可读性的必要条件；永远保留纯文本可读版（Matrix 强制 body fallback 同理）。
- Outbound 每渠道一个 renderer 降级渲染：Telegram → HTML parse_mode 或 entities（MarkdownV2 有 18 个必转义字符、解析失败整条 400，绕开）；Discord → 近似透传、剔除表格；邮件 → HTML+plain multipart 双轨。
- 出站可靠性做成 transport 层拦截器（grammY auto-retry 模式）：429 按 retry_after 等、5xx 指数退避（3s 起步）、401/409 致命错上抛停用 adapter。
- 重连范本抄 Discord gateway 规范：心跳未 ACK 即视为僵尸连接立断、优先 Resume（带序列号补漏）、不行再全量 Identify + 指数退避——顺便统一 Telegram"重放"与 Discord"丢失"的不一致掉线语义。

### 2.6 明确不做（过度设计）

Matrix appservice 协议层（注册文件/双 token/幽灵用户/double puppeting）、进程内消息队列中间件、自动跨渠道身份合并、BF 中间件洋葱与卡片体系、Novu 式每渠道模板、完整 mdast AST 管线（3-4 渠道先"交集自律 + 每渠道一个转换函数"）。

## 3. 升级步骤（分阶段，每步可独立验证）

### Phase 0 — 诚实化 + 白捡（小）
1. 修 FEATURES.md:211-214 与 zh-CN「双向通信」文案过度承诺 → 验证：文案与实际能力一致
2. merge 79434d0（media + routeTelegramMessage seam + 13 测试）→ 验证：clean merge、全测试绿
3. allowlist 空集默认 deny（逻辑反转 + 设置页提示文案）→ 验证：空 allowlist 下陌生消息被丢弃 + 测试

### Phase 1 — 回复回路闭合（v0.4.0 核心）
1. `assistantReply.ts` 加 `onReplyComplete(source, finalText, target)` 回调（seam 在 :461-499 区间，source 与定稿文本同作用域）
2. bridge controller 接回调 → 调既有 `replyTo`，经降级 renderer 发回原渠道 → 验证：Telegram/Discord 发问，伴侣回复回到渠道
3. 修连带 bug：busy 时桥消息排队或明确反馈（不再无声丢弃）；手动回复假成功（replyTo 返回值判定）；权限门 `needs_confirmation` 给出消费者或简化掉
4. 出站回声去重 + 入站 update_id 幂等（防自言自语、防重连重放）
5. gateway 可测试化：仿 minecraftGateway 模式注入 net.fetch/WebSocket（收口仅 3 处），桥链路上测试

### Phase 2 — ChannelAdapter 收敛
1. 抽 `ChannelAdapter` 接口 + 中心 Hub，三对孪生文件合并；webhook 渠道补"进 chat"能力
2. 消息 ID 映射表落地（回复引用从此查表换算）
3. `broadcastToChannels` 与所有 send 收口到 Hub 同一出口（权限门、节流统一生效）
4. 配对码安全流程替换裸 allowlist 配置
5. 掉线语义统一（Resume/退避状态机共享）

### Phase 3 — 邮件渠道（v0.5+，全绿地）
抄 Delta Chat：1:1 按 From 地址聚合会话（不靠 thread）、陌生来信进 Request 确认区（= 邮件版配对）、保守 quote-strip（simplify.rs 策略：剥完只剩空白则回退原文）、IMAP IDLE（RFC 2177，29 分钟重发）+ 轮询兜底；Gmail 走 IMAP IDLE 或 history.list 增量（push 通知假设有后端服务器，桌面应用不现实）。**安全：From allowlist 必须叠加 Authentication-Results 头校验**（From 可伪造，仅 DMARC alignment 挡 exact-domain spoofing）。不做：Autocrypt/SecureJoin、群聊信令、JWZ 全量线程树。

### Phase 1.5（OpenClaw/Hermes 对比调研补充）—— 前三项已交付（2026-06-11）

- ✅ 语音回信 `inbound` 模式：Telegram 语音条现在会被下载并经用户配置的 API STT 转写进伴侣对话（无 API STT 时回退到原 announce-only）；`telegramVoiceReplyMode: off/always/inbound`（inbound=对方发语音才回语音），旧布尔设置自动迁移
- ✅ webhook 安全加固（ClawJacked 教训）：Bearer 改 `timingSafeEqual` 常数时间比对、失败限速（10 次/分钟滑动窗，无任何来源豁免——本机即攻击面）、移除全部 CORS 头（浏览器预检直接失败，网页 JS 进不来）
- ✅ 本地免费 TTS：sherpa-onnx VITS（MeloTTS 中英，~165MB 可选模型走现有模型安装器）；`local-tts` provider 同时接入桌面播报流水线（PCM）与 audio:synthesize（WAV）；零 Key 零计费。**限制：输出 wav，Telegram 语音条（须 mp3/ogg/m4a）会自动降级为纯文字；Discord 音频附件可用**
- ⏳ HEARTBEAT_OK 式静默剥离（自主引擎"无事不发声"机制化）——未做，属自主引擎改造
